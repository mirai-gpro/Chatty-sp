分析結果：コンシェルジュモードが検索に辿りつけない根本原因
旧版（gourmet-support）で機能していた仕組み
旧RESTモードでは：

ユーザーが「新宿で接待」と発言 → /api/chat にPOST
AIがJSON形式で応答 → message(テキスト) + shops(配列)
会話履歴がサーバー側のRAMセッション（SupportSession）に保存される
次のターンで「焼肉がいい」→ 過去の履歴込みでGemini REST APIに送信
Geminiは全ターンの履歴を見て条件が揃ったか判断できた
「お探ししますね」→ shops配列にデータを入れてJSON返却
要するに：RESTモードではGemini REST APIが全会話履歴を毎回受け取っていたので、短期記憶は完璧に機能していた。

LiveAPIで壊れた理由
live_api_handler.py を見ると：

LiveAPIはステートフルなWebSocket接続 — 会話中は1セッション内でGeminiが記憶を保持する → ここは問題ない
しかし、再接続が頻繁に起きる — 累積800文字 / 長い発話500文字 / 発言途切れ → needs_reconnect = True
再接続時のコンテキスト注入が不十分：

これがシステムプロンプトの末尾に追加されるだけ。

問題の核心：コンシェルジュのプロンプト（concierge_ja.txt）がLiveAPIのシステムプロンプトに含まれていない
live_api_handler.py:62-94 の LIVEAPI_CONCIERGE_SYSTEM はわずか30行程度の簡素なプロンプト。concierge_ja.txt の537行の詳細仕様（ステップ1〜4のヒアリング順序、短期記憶ルール、業態別制御等）は一切含まれていない。

さらに、再接続のたびにGeminiは「新しいセッション」として始まる — _get_context_summary()の要約は「user: 新宿で...」「ai: 承知しました...」のような断片的テキストでしかなく、短期記憶のセマンティクス（何が確認済みで何が未確認か）が失われる。
具体的な症状の説明
ターン1: ユーザー「接待で使いたい」 → AI「どのエリアですか？」
ターン2: ユーザー「六本木で」 → AI「和食と洋食どちらが？」
  ← ここで累積文字数が500超え or 発言途切れ判定 → 再接続発生
ターン3: （再接続後）Geminiは要約を見るが「条件が何で、何が未確認か」がわからない

  → ユーザー「さっき言った…」 → AI「承知しました。エリアは？」 ← 無限ループ

    }
    for kw, purpose in purpose_keywords.items():

    # ---- ジャンル検出 ----

        '和食': '和食', '洋食': '洋食', 'イタリアン': 'イタリアン',

    if party_match:
        self.short_term_memory['party_size'] = f"{party_match.group(1)}名"
    elif '二人' in user_text or 'ふたり' in user_text:
        self.short_term_memory['party_size'] = '2名'

    # ---- 雰囲気検出 ----

ヒアリングステップの自動更新
def _update_hearing_step(self):
    """確定条件に基づいてヒアリングステップを更新"""
    m = self.short_term_memory
    
    if not m['area'] and not m['purpose']:
        self.hearing_step = 1  # まだステップ1
    elif not m['cuisine'] and not m['party_size']:
        self.hearing_step = 2  # ステップ2へ
    elif not m['budget']:
        self.hearing_step = 3  # ステップ3へ
    elif not m['date']:
        self.hearing_step = 4  # ステップ4へ
    else:
        self.hearing_step = 5  # 全条件確定 → 検索可能

def _get_context_summary(self) -> str:
    """
    再接続時のコンテキスト注入。
    チャットログの断片ではなく、構造化された状態を注入する。
    """
    parts = []

    # ===== 1. 確定済み条件（短期記憶） =====
    m = self.short_term_memory
    confirmed = []
    unconfirmed = []

    condition_labels = {
        'area': 'エリア', 'purpose': '利用目的',
        'cuisine': '料理ジャンル', 'atmosphere': '雰囲気',
        'party_size': '人数', 'budget': '予算', 'date': '日時',
    }
    for key, label in condition_labels.items():
        if m[key]:
            confirmed.append(f"  - {label}: {m[key]}（確認済み）")
        else:
            unconfirmed.append(f"  - {label}: 未確認")

    parts.append("【ユーザーの条件（短期記憶）】")
    if confirmed:
        parts.append("＜確定済み＞")
        parts.extend(confirmed)
    if unconfirmed:
        parts.append("＜未確認＞")
        parts.extend(unconfirmed)

    # ===== 2. 禁止事項 =====
    if confirmed:
        parts.append("")
        parts.append("【禁止】上記の確定済み条件を再度質問してはならない。")

    # ===== 3. 次にすべきこと =====
    step_instructions = {
        1: "ステップ1：エリアと利用目的を質問してください。",
        2: "ステップ2：料理ジャンル・雰囲気・人数を質問してください。",
        3: "ステップ3：予算を質問してください。",
        4: "ステップ4：日時を質問してください。",
        5: "全条件が揃っています。「お探ししますね」と言って検索を開始してください。",
    }
    parts.append(f"\n【現在のステップ】{step_instructions.get(self.hearing_step, '')}")

    # ===== 4. 直近の会話（最小限、参考用） =====
    if self.conversation_history:
        recent = self.conversation_history[-4:]  # 直近4ターンだけ
        parts.append("\n【直近の会話（参考）】")
        for h in recent:
            role_label = "ユーザー" if h['role'] == 'user' else "AI"
            parts.append(f"  {role_label}: {h['text'][:100]}")

    return "\n".join(parts)

handleShopSearchFromLiveAPI の L586-595:

コミット 6e32fed の分析結果
フロー:

LiveAPI AI応答でトリガーキーワード検知 → _shop_search_pending フラグを立てる → session_loop 終了

_restart_live_with_message() で通常会話に復帰
「ショップカード提示まで辿りつけてない。途中でクリアされてる？」の原因候補:

1. _fetch_shop_data() がデータを返せていない可能性が高い
_shop_search_internal() は SupportAssistant.process_user_message() を呼んでいるが、LiveAPI経由の会話履歴は SupportSession に保存されていない。_build_search_request() で構築した user_request テキストだけでは、Gemini REST APIがショップJSON を生成しない可能性がある（会話コンテキスト不足）。

→ この問題はその後の d7b1528 fix: inject LiveAPI conversation history で修正を試みている。

2. shop_search_result の emit は通っているが、フロントで受け取れていない可能性
core-controller.ts のリスナーは shop_search_result に置き換え済みだが、コンシェルジュモードでは concierge-controller.ts が initSocket() をオーバーライドして super.initSocket() を呼んでいるので、リスナー自体は登録されている。ここは問題なさそう。

3. 最も怪しい: _handle_shop_search 内で例外が起きて shop_search_result emit に到達していない
asyncio.to_thread() でコールバックを呼んでいるが、SupportAssistant 内部でエラーが起きると None が返り、「ショップ見つからず」ルートに入る → カード表示されない。

結論: ショップカード表示に辿り着けない原因は _fetch_shop_data() → _shop_search_internal() のREST検索がショップデータを返せていない ことが最も可能性が高いです。LiveAPI会話のコンテキストがSupportSessionに入っていないため、REST API側が「ショップ検索」と認識できず、通常の会話応答を返している（shops が空）。
