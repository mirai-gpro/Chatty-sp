# Cloud Run ビルド再現性問題 — 運用メモ

## 発生した問題

**同じコミット（Git差分ゼロ）でも、Cloud Runに再デプロイすると挙動が変わる。**

### 具体例（2026-03-30）
- コミット `7290255` のCloud Runリビジョン `chatty-sp-base-00060-dn9` → リップシンク正常
- 同じバックエンドコードで再デプロイ（revert後） → リップシンク壊れる
- GCPコンソールで旧リビジョンに戻すと正常に復帰

### 原因
`requirements.txt` に `pdfplumber==0.11.6` を追加していた。
Cloud Run上では不要なパッケージだが、その依存ツリーがビルド時に解決される際に、他のパッケージ（scipy、numpy等）のバージョンや依存関係に影響を与え、A2Eリップシンクの動作が壊れた。

## 教訓

### 1. Git revert ≠ Cloud Run復旧
- Gitのファイル差分がゼロでも、Cloud Buildが再ビルドすると前回と同じコンテナになる保証はない
- 障害時の第一手は **Git revert + 再ビルドではなく、GCPコンソールで正常だった旧リビジョンにトラフィックを戻す**
- Cloud Runのリビジョンはimmutable（不変）なので、旧リビジョンは当時の実体そのまま

### 2. requirements.txt には本番に必要なものだけ
- 開発ツール用の依存は `requirements-dev.txt` に分離する
- Cloud Runのビルドで使う `requirements.txt` には本番実行に必要なパッケージのみ記載
- 不要なパッケージの依存ツリーが予期せぬ副作用を引き起こす

### 3. 依存追加時はリップシンク確認必須
- `requirements.txt` にパッケージを追加した場合、デプロイ後に必ずリップシンクの動作確認を行う
- 特にscipy、numpy等の数値計算ライブラリに影響を与える可能性のあるパッケージは要注意

## ファイル構成

```
chatty-base/
  requirements.txt      ← Cloud Run本番用（必要最小限のみ）
  requirements-dev.txt  ← 開発ツール用（pdfplumber等、ローカルで使うもの）
```

## 障害時の対応手順

1. **即座**: GCPコンソール → Cloud Run → リビジョン → 正常だった旧リビジョンにトラフィック100%
2. **原因特定**: 旧リビジョンと新リビジョンの差分を確認（image digest、env vars、build logs）
3. **修正**: 原因を特定してからコード修正 → 新しいコミットでデプロイ
4. **確認**: デプロイ後にリップシンクを含む全機能を確認
